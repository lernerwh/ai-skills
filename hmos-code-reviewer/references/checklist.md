# 鸿蒙代码检视 Checklist

本文档定义了鸿蒙(HarmonyOS)/ArkTS 代码 AI 检视的检查项清单。

---

## 1. 代码风格与规范

### 1.1 命名规范
- [ ] 类名使用 PascalCase（如 `ChatDetail`、`MessageDataSource`）
- [ ] 变量和函数使用 camelCase（如 `currentChatId`、`loadMessages`）
- [ ] 常量使用 UPPER_SNAKE_CASE（如 `MAX_RETRY_COUNT`）
- [ ] 私有成员以 `_` 开头或使用 `private` 修饰符
- [ ] 布尔变量使用 `is`、`has`、`should` 等前缀（如 `isLoading`、`hasError`）
- [ ] 接口/类型命名清晰表达用途

### 1.2 代码格式
- [ ] 缩进使用空格或 Tab 保持一致（ArkTS 推荐 2 空格）
- [ ] 每行代码长度不超过 120 字符
- [ ] 函数之间保持适当空行
- [ ] import 语句按类型分组（标准库、第三方库、本地模块）
- [ ] 文件末尾有且只有一个换行符

### 1.3 注释规范
- [ ] 复杂逻辑有清晰的注释说明
- [ ] 公共函数/方法有 JSDoc 风格的文档注释
- [ ] 注释内容与代码保持同步，无过期注释
- [ ] 避免无意义的注释（如 `// 初始化变量`）
- [ ] TODO 注释包含负责人或时间信息

---

## 2. 安全性问题

### 2.1 输入验证
- [ ] 用户输入经过验证和清理
- [ ] 避免直接拼接 SQL 语句，使用参数化查询
- [ ] 外部数据（网络、文件）使用前进行校验
- [ ] 敏感操作有权限检查

### 2.2 数据安全
- [ ] 敏感数据（密码、Token）不明文存储
- [ ] 日志中不输出敏感信息
- [ ] 使用安全的数据存储方式（如 `SecurityLevel.S1` 以上）
- [ ] 临时数据及时清理

### 2.3 空值安全
- [ ] 可能为 null/undefined 的值有空值检查
- [ ] 使用可选链操作符 `?.` 访问可能为空的对象属性
- [ ] 使用空值合并操作符 `??` 提供默认值
- [ ] 避免隐式的类型转换可能导致的问题

### 2.4 异常处理
- [ ] 使用 try-catch 包装可能抛出异常的代码
- [ ] catch 块中正确处理错误，不只是 `console.log`
- [ ] 异常信息足够详细，便于定位问题
- [ ] 关键资源在 finally 中释放

---

## 3. 性能问题

### 3.1 渲染性能
- [ ] 避免在 build 方法中执行耗时操作
- [ ] 使用 `@State`、`@Prop` 等装饰器合理管理状态
- [ ] 列表渲染使用 `LazyForEach` 而非 `ForEach`（大数据量时）
- [ ] 避免不必要的状态更新触发重新渲染
- [ ] 使用 `cachedCount` 优化长列表性能

### 3.2 内存管理
- [ ] 及时释放不再使用的资源
- [ ] 避免内存泄漏（事件监听器未移除、定时器未清除）
- [ ] 大对象使用后置为 null
- [ ] 图片等资源合理管理生命周期

### 3.3 异步处理
- [ ] 耗时操作使用 async/await
- [ ] 并行任务使用 Promise.all 提高效率
- [ ] 避免回调地狱，使用 async/await 替代嵌套回调
- [ ] 合理设置超时时间

### 3.4 数据操作
- [ ] 数据库查询有适当的索引
- [ ] 批量操作使用事务
- [ ] 避免循环中执行数据库/网络操作
- [ ] 大数据量操作考虑分页/分批处理

---

## 4. 可维护性

### 4.1 代码组织
- [ ] 单个文件不超过 500 行（复杂组件可适当放宽）
- [ ] 单个函数不超过 50 行
- [ ] 文件职责单一，避免"上帝类"
- [ ] 相关功能按模块组织目录结构

### 4.2 函数设计
- [ ] 函数职责单一（单一职责原则）
- [ ] 函数参数不超过 4 个，超过时考虑使用对象
- [ ] 避免深层嵌套（超过 3 层）
- [ ] 函数有明确的返回值

### 4.3 代码重复
- [ ] 重复代码抽取为公共函数/组件
- [ ] 使用继承或组合减少代码重复
- [ ] 常量、配置统一管理

### 4.4 依赖管理
- [ ] 避免循环依赖
- [ ] 依赖关系清晰，层次分明
- [ ] 使用依赖注入降低耦合度

---

## 5. 错误处理

### 5.1 错误捕获
- [ ] 所有 async 函数有 try-catch
- [ ] 网络请求有错误处理
- [ ] 文件操作有错误处理
- [ ] 数据库操作有错误处理

### 5.2 错误恢复
- [ ] 关键操作有重试机制
- [ ] 失败后提供用户友好的提示
- [ ] 有适当的降级策略
- [ ] 错误状态有 UI 反馈

### 5.3 日志记录
- [ ] 关键操作有日志记录
- [ ] 错误日志包含足够的上下文信息
- [ ] 使用合适的日志级别（debug/info/warn/error）
- [ ] 生产环境不输出过多调试日志

---

## 6. ArkUI/ArkTS 特定问题

### 6.1 组件设计
- [ ] 使用 `@Component` 装饰器定义组件
- [ ] 状态变量使用正确的装饰器（`@State`、`@Prop`、`@Link` 等）
- [ ] `@Builder` 方法用于可复用的 UI 片段
- [ ] 组件间通信使用 `@Provide`/`@Consume` 或 `@Watch`

### 6.2 状态管理
- [ ] 合理使用 `@State`、`@Prop`、`@Link`、`@ObjectLink`
- [ ] 避免过度使用 `@StorageLink`/`@StorageProp`
- [ ] 状态更新是响应式的
- [ ] 复杂状态考虑使用 ViewModel 模式

### 6.3 UI 性能
- [ ] 使用 `List` + `LazyForEach` 渲染长列表
- [ ] 图片使用适当的缓存策略
- [ ] 动画使用硬件加速（`animateTo`）
- [ ] 避免过度绘制

### 6.4 生命周期
- [ ] 正确使用 `aboutToAppear`/`aboutToDisappear`
- [ ] 在 `aboutToDisappear` 中清理资源
- [ ] 页面跳转正确管理导航栈

### 6.5 ArkTS 语法
- [ ] 正确使用 ArkTS 的类型系统
- [ ] 避免使用 TypeScript 不支持的特性
- [ ] 使用 `@Observed`/`@ObjectLink` 处理嵌套对象
- [ ] 正确使用 `HashMap`、`ArrayList` 等 ArkTS 集合类型

---

## 7. 业务逻辑

### 7.1 逻辑正确性
- [ ] 业务逻辑边界条件处理正确
- [ ] 状态转换符合业务规则
- [ ] 数据校验符合业务需求
- [ ] 并发场景有适当的锁/同步机制

### 7.2 用户体验
- [ ] 加载状态有 UI 指示
- [ ] 操作有适当的反馈（Toast、Dialog）
- [ ] 错误信息对用户友好
- [ ] 关键操作有确认机制

---

## 8. 测试相关

### 8.1 可测试性
- [ ] 代码结构便于单元测试
- [ ] 依赖可以 mock
- [ ] 纯函数与副作用分离

### 8.2 测试覆盖
- [ ] 关键业务逻辑有单元测试
- [ ] 边界条件有测试用例
- [ ] 异常路径有测试覆盖

---

## 严重程度映射指南

| 问题类型 | 建议严重程度 |
|---------|-------------|
| 安全漏洞、数据丢失风险 | Critical |
| 空指针异常、资源泄漏、未处理异常 | High |
| 代码重复、复杂度过高、性能问题 | Medium |
| 命名不规范、注释缺失、格式问题 | Low |

---

## 使用说明

AI 检视时，针对每个检查项：
1. **逐项检查**：不遗漏任何检查项
2. **记录依据**：标注发现问题时依据的检查项
3. **客观评估**：严重程度按上述指南评估
4. **精确定位**：提供具体的文件路径和行号

# 鸿蒙代码 AI 检视技能

专业的 HarmonyOS/ArkTS 代码 AI 检视工具，基于预定义的 checklist 进行深度代码分析，生成结构化的检视报告。

## 工作流程

```
输入(文件/目录) → 制定检视计划 → 拆分检视任务 → 并行AI检视 → 汇总CSV报告
```

## 步骤 1: 接收输入并分析

1. **确定检视范围**：用户指定文件或目录
2. **扫描目标文件**：支持 `.ets`、`.ts` 文件
3. **估算文件数量和大小**：决定是否需要拆分任务

## 步骤 2: 制定检视计划

根据检视范围制定任务计划：

### 小规模检视（≤5个文件，每个文件≤500行）
- 单任务检视，直接执行

### 中等规模检视（6-20个文件）
- 按模块/目录拆分为2-4个任务
- 每个任务检视一个逻辑单元

### 大规模检视（>20个文件或大文件）
- **必须拆分**为多个子任务，每个任务检视3-5个文件
- 目标：每个任务的上下文控制在合理范围内
- 按功能模块或目录结构拆分

**计划输出格式**：
```
## 检视计划

### 任务1: [模块名称]
- 文件: file1.ets, file2.ets
- 预计行数: XXX行

### 任务2: [模块名称]
- 文件: file3.ets, file4.ets, file5.ets
- 预计行数: XXX行

...
```

## 步骤 3: 读取 Checklist

在执行检视前，**必须**读取检视清单：

```
使用 Read 工具读取: references/checklist.md
```

Checklist 包含以下检视维度：
- 代码风格与规范
- 安全性问题
- 性能问题
- 可维护性
- 错误处理
- ArkUI/ArkTS 特定问题

## 步骤 4: 执行 AI 检视

### 4.1 检视任务执行

对于每个检视任务，启动一个 subagent 执行 AI 检视：

```
task(
  category="unspecified-high",
  load_skills=[],
  run_in_background=true,
  description="检视[模块名]代码",
  prompt="""
  ## 任务
  对以下鸿蒙代码文件进行 AI 检视，基于 checklist 中的检查项逐项审查。
  
  ## 检视文件
  - [文件路径1]
  - [文件路径2]
  
  ## Checklist 参考
  [从 references/checklist.md 读取的关键检查项]
  
  ## 输出要求
  生成 CSV 格式的检视报告，包含以下列：
  - 问题描述: 具体的问题描述
  - 文件路径: 代码文件完整路径
  - 行号范围: 如 "42" 或 "100-120"
  - 严重程度: Critical / High / Medium / Low
  - 检视依据: 对应 checklist 中的检查项
  
  ## 注意事项
  1. 只报告真实存在的问题，不要过度检视
  2. 提供具体的行号，便于定位
  3. 严重程度要合理评估
  4. 对于 ArkTS 特有的问题要特别关注
  """
)
```

### 4.2 并行执行

- 多个检视任务**同时启动**，并行执行
- 每个任务独立生成 CSV 片段
- 任务 ID 需要记录，用于后续收集结果

## 步骤 5: 收集检视结果

等待所有 subagent 完成后，使用 `background_output` 收集每个任务的检视结果：

```
background_output(task_id="[task_id]")
```

## 步骤 6: 汇总检视报告

### 6.1 合并所有 CSV 报告

将所有任务的 CSV 结果合并为一份最终报告：

**CSV 格式规范**：
```csv
问题描述,文件路径,行号范围,严重程度,检视依据
"变量未使用","/path/to/file.ets","15","Low","代码风格-无用代码"
"空指针风险","/path/to/file.ets","42-45","High","安全性-空值检查"
```

### 6.2 保存报告文件

将最终报告保存到用户指定位置或当前工作目录：

```
检视报告保存路径: [工作目录]/code_review_report_[时间戳].csv
```

## 步骤 7: 输出检视总结

生成简明的检视总结：

```
## 检视总结

### 统计
- 检视文件数: X 个
- 发现问题数: Y 个
  - Critical: X 个
  - High: X 个
  - Medium: X 个
  - Low: X 个

### 主要问题类型
1. [问题类型1]: X 个
2. [问题类型2]: X 个
3. [问题类型3]: X 个

### 建议优先处理
- [最严重的问题摘要]

### 报告文件
- CSV报告: [文件路径]
```

## 严重程度定义

| 级别 | 含义 | 示例 |
|------|------|------|
| **Critical** | 必须修复，可能导致崩溃或安全漏洞 | 空指针解引用、SQL注入、敏感数据泄露 |
| **High** | 应该修复，影响功能或稳定性 | 资源泄漏、未处理的异常、性能瓶颈 |
| **Medium** | 建议修复，影响代码质量 | 代码重复、复杂度过高、命名不规范 |
| **Low** | 可选修复，优化建议 | 注释缺失、格式问题、轻微的代码异味 |

## 检视原则

1. **准确性优先**：只报告确定存在的问题，避免误报
2. **具体可操作**：提供精确的行号和修复建议
3. **上下文相关**：考虑鸿蒙/ArkTS 的特定场景
4. **分级合理**：严重程度评估要客观

## 使用示例

### 示例1: 检视单个文件
```
用户: 检视一下 features/home/src/main/ets/pages/Chat/ChatDetail.ets
```

### 示例2: 检视目录
```
用户: 对 features/home/src/main/ets/viewmodel 目录下的代码进行检视
```

### 示例3: 检视整个项目
```
用户: 检视整个项目的代码质量
```

## 注意事项

1. **文件读取**：检视前需要完整读取目标文件内容
2. **上下文控制**：大文件/多文件时务必拆分任务
3. **并行效率**：合理并行可大幅提升检视速度
4. **结果验证**：汇总前检查每个任务是否成功完成
5. **报告完整性**：确保所有问题都被记录到最终报告
